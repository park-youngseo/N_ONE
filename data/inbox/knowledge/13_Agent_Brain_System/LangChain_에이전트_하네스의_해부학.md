## [섹션 1: 하네스(Harness)의 정의]

> "모델이 지능이라면, 하네스는 그 지능을 유용하게 만드는 시스템이다."

- **정의**: 모델 자체를 제외한 모든 코드, 설정, 실행 로직.
- **포함 요소**:
  - 시스템 프롬프트 (System Prompts)
  - 도구, 스킬, MCP 및 그 설명들
  - 번들 인프라 (파일 시스템, 샌드박스, 브라우저)
  - 오케스트레이션 로직 (서브에이전트 스폰, 핸드오프, 모델 라우팅)
  - 결정론적 실행을 위한 훅/미들웨어 (Lint 체크, 컴팩션 등)

## [섹션 2: 하네스 설계의 핵심 컴포넌트]

### 1. 파일 시스템 (Filesystems) — 기초 primitive
- **목적**: 영구 저장소 제공, 컨텍스트 용량 초과 정보 오프로드, 세션 간 작업 유지.
- **장점**: 
  - 에이전트가 데이터, 코드, 문서를 읽는 워크스페이스 확보.
  - 중간 출력을 저장하여 단일 세션보다 오래 지속되는 상태 유지.
  - 에이전트 팀 간의 협업 표면(Shared Ledger) 역할.
- **Git 활용**: 버전 관리를 통해 오류 복구 및 실험 브랜칭 가능.

### 2. Bash + Code (범용 도구)
- **목적**: 사전에 정의되지 않은 문제도 자율적으로 해결.
- **기능**: 모델이 스스로 도구를 설계하고 코드를 실행하여 문제를 해결하게 함.

### 3. 샌드박스 (Sandboxes)
- **목적**: 안전한 실행 및 검증.
- **기능**: 격리된 환경에서 코드 실행, 의존성 설치, 로그 분석. 스케일링을 위해 필요 시 생성/삭제.

### 4. 컨텍스트 로트(Context Rot) 방지 기술
- **컴팩션(Compaction)**: 컨텍스트 창이 찰 때 지능적으로 요약하여 작업 지속.
- **도구 출력 오프로딩**: 너무 긴 도구 출력은 파일 시스템으로 보내고 토큰 절약.
- **스킬(Skills)**: 수많은 도구 중 필요한 것만 동적으로 로드하여 초기 성능 저하 방지.

---

## [섹션 3: 장기 자율 실행 (Long Horizon Execution)]

복잡한 작업을 오랫동안 수행하기 위한 하네스 패턴:
- **Ralph Loops**: 모델의 종료 시도를 가로채서 깨끗한 컨텍스트 창에 원래 프롬프트를 재주입, 완료될 때까지 강제 지속.
- **계획 및 자기 검증(Self-verification)**: 목표를 분해하고, 각 단계 완료 후 테스트 슈트를 실행하여 스스로 오류 수정.

---

## [섹션 4: 핵심 인사이트 — 하네스 엔지니어링]

1. **프롬프트 엔지니어링 vs 하네스 엔지니어링**:
   - 프롬프트: 단일 대화 턴 최적화.
   - 하네스: 수 시간 동안 지속되는 자율 실행 환경 설계.
2. **모델과의 공동 진화**: 최신 모델(Claude Code, Codex 등)은 이미 특정 하네스 환경에 맞춰 포스트 트레이닝됨.
3. **최적화의 여지**: 같은 모델이라도 하네스 설계에 따라 성능 차이가 극명함 (Terminal Bench 2.0 사례).

## [원본 텍스트 요약 — 무손실]

> **[AX 하네스 준수]** 아래는 원본의 핵심 논리를 보존한 정제본입니다.

- **모델의 한계**: 텍스트 입출력은 잘하나 상태 유지, 코드 실행, 실시간 지식 접근, 환경 설정 등은 불가.
- **해결책**: 하네스가 모델을 감싸서 이러한 기능을 수행하게 함 (Agentic UI/UX).
- **파일 시스템**: 에이전트에게 "컴퓨터"를 주는 첫걸음. `AGENTS.md` 같은 표준을 통해 지속적 학습(Continual Learning) 구현.
- **자기 개선**: 에이전트가 자신의 실행 흔적(Traces)을 분석하여 하네스 수준의 실패 모드를 수정하는 단계로 진화 중.
