## Purpose
수강생들이 자신의 출결 상태와 대본 연습 진도를 직관적으로 확인하여 학습 몰입도를 높이고 체계적인 관리를 가능하게 합니다. `--ultrathink` 적용으로 다수 접속 환경에서의 실시간 진도 동기화 및 데이터 일관성 유지를 목표로 고도화되었습니다.

## Background
기존 `0.1.0` 스펙에서는 REST API 기반의 단방향 데이터 조회를 설계했으나, 학원 내 단체 그룹 연습이나 강사-학생 간 실시간 상호작용 상황에서는 폴링(Polling)으로 인한 서버 부하 및 상태 불일치 문제가 발생할 수 있음을 분석했습니다. 이를 해결하기 위해 WebSocket과 Redis를 도입하여 상태 관리를 실시간으로 격상합니다.

## Requirements

### Ubiquitous
- 시스템은 수강생의 월별 출결 현황(출석, 지각, 결석)을 달력 형태로 표시해야 합니다.
- 시스템은 수강 중인 대본별 연습 진도율(%)을 프로그레스 바로 표시해야 합니다.
- 시스템은 Redis 캐싱 계층을 통해 대시보드 진입 시 0.5초 이내에 초기 렌더링 데이터를 반환해야 합니다.

### Event-Driven
- WHEN 수강생이 특정 장면의 연습 상태(읽기, 암기, 분석, 리허설)를 완료로 체크할 때, THEN 시스템은 데이터베이스를 갱신함과 동시에 Message Broker(Redis Pub/Sub)를 통해 이벤트를 발행해야 합니다.
- WHEN 대본 진도 업데이트 이벤트가 Message Broker에 발행될 때, THEN 시스템은 WebSocket을 통해 해당 대본 화면을 보고 있는 모든 관련 클라이언트(강사 및 소속 수강생)에게 새로운 진행률 데이터를 즉각 푸시(Push)해야 합니다.
- WHEN 수강생이 대시보드에 접속할 때, THEN 시스템은 가장 최근에 업데이트된 대본의 진도를 요약하여 메인에 노출해야 합니다.

### Unwanted
- IF 출결 데이터 로드에 실패할 경우, THEN 시스템은 사용자에게 오류 메시지를 표시하고 '다시 시도' 버튼을 제공해야 합니다.
- IF WebSocket 연결이 끊어질 경우, THEN 시스템은 지수 백오프(Exponential Backoff) 알고리즘을 사용하여 백그라운드에서 재연결을 시도해야 하며, 연결 단절 중 발생한 오프라인 상태의 체크 이벤트는 큐에 적재되었다가 복구 시 일괄 동기화(Sync) 되어야 합니다.

### Optional
- WHERE '다크 모드'가 활성화된 경우, THEN 시스템은 연극 무대 느낌의 테마를 적용해야 합니다.

## Acceptance Criteria
- [ ] 수강생이 로그인을 성공하면 메인 대시보드에서 자신의 출석률을 확인할 수 있다.
- [ ] 대본 상세 페이지에서 장면별 체크리스트를 클릭하면 상단의 진행률 바가 실시간으로 변경된다.
- [ ] 브라우저 탭 A에서 진도를 체크했을 때, 동일한 계정으로 접속된 브라우저 탭 B에서 새로고침 없이 프로그레스 바가 즉시(1초 이내) 갱신된다. (WebSocket 검증)
- [ ] 모바일 기기에서도 달력과 프로그레스 바가 깨짐 없이 정상적으로 표시된다.
